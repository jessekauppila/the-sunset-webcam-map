# ML Diagnostics Findings

Date: April 2026
Context: First diagnostic pass after professor consultation.
Plots generated by: `python3 ml/plot_diagnostics.py --all`

---

## What we looked at

7 completed experiment runs were analyzed:
- 2 early baseline runs (tiny data, ~22 samples — ignore these)
- 4 full binary classification runs (3,284 train samples each)
- 1 regression run

The three diagnostic questions from the professor:
1. How are labels distributed? (Is data clustered in the middle?)
2. What do the loss curves look like? (Are we overfitting?)
3. What is the class balance? (Are great sunsets underrepresented?)

---

## Finding 1: Significant class imbalance

Across all full runs, the training set had:

| Class | Count | Share |
|-------|-------|-------|
| Negative (not a great sunset) | 2,638 | 80% |
| Positive (great sunset) | 646 | 20% |
| Ratio | 4.1:1 | |

**What this means:** Without correcting for imbalance, a model that always
predicts "not great" would be right 80% of the time. Accuracy alone is a
useless metric here.

**What we did:** `class_weighting: balanced` was used, which upweights
positives ~4x in the loss (effective weights: [0.62, 2.54]). This is the
right approach.

**What remains:** We do not know yet whether 646 positive examples is
enough, or whether we need more labeled great sunsets. The label
distribution plots show where the raw ratings cluster — check
`plots/label_distribution.png` inside each run directory.

## Finding 2: Both runs overfit — more epochs will not help

The two best binary runs tell the same story:

### `v2_mild_crop_balanced` (random resized crop, 10 epochs)

| Epoch | Train loss | Val loss | Val F1 |
|-------|-----------|----------|--------|
| 1 | 0.381 | 0.418 | 0.725 |
| 4 | 0.163 | **0.287** ← val loss minimum | 0.800 |
| 8 | 0.091 | 0.316 | 0.802 |
| 10 | 0.075 | 0.459 | **0.832** ← best F1 |

Train loss drops smoothly the whole time. Val loss hit its best at epoch 4
(0.287) then drifted up to 0.459 by epoch 10 — 60% worse. F1 still
improved because the classification threshold still works, but the model's
probability estimates are degrading. This is the definition of overfitting.

### `v2_no_crop_balanced` (no crop, 10 epochs)

| Epoch | Train loss | Val loss | Val F1 |
|-------|-----------|----------|--------|
| 1 | 0.387 | 0.339 | 0.782 |
| **2** | 0.235 | **0.315** | **0.834** ← peaked here |
| 3 | 0.178 | 0.526 | 0.748 |
| 7 | 0.089 | 0.572 | 0.803 |
| 10 | 0.069 | 0.514 | 0.743 |

This run peaked at epoch 2. By epoch 7, val loss doubled (0.315 → 0.572).
By epoch 10, F1 dropped back to 0.743 — worse than where it started.
Running more epochs here actively made the model worse.

**Conclusion: more epochs, with the current setup, will not produce better
results.** The model learns the training set and the validation loss
diverges. The ceiling is a data and regularization problem, not an epochs
problem.

---

## Finding 3: What 0.83 F1 actually means

F1 of 0.83 is decent for a first-pass model. It is not the ceiling. Typical
well-tuned image classifiers on this kind of task reach 0.88–0.92 with
better data and regularization.

But F1 is a balance between precision and recall. For the "archive great
sunsets" use case, **recall** is the more important metric — missing a great
sunset is worse than occasionally capturing a mediocre one. To know whether
0.83 is good enough, you need to look at the confusion matrix in
`eval/eval_report.json`:

- `fn` (false negatives) = great sunsets that were missed
- `fp` (false positives) = mediocre sunsets that were captured
- `recall` = fraction of actual great sunsets that were caught

A model with F1 of 0.83 but low recall is the wrong trade-off for this
application.

---

## Root causes of overfitting

In priority order:

1. **Val set is small (723 images).** A small val set means loss bounces a
   lot — a few hard images can swing the number significantly. This makes
   it hard to know when the model has truly peaked.

2. **Label noise likely.** Ratings cluster around 2.5–3.5. Images near the
   binary threshold (e.g. rating 3.0) may be labeled positive in one export
   and negative in another depending on exact threshold. The model sees
   visually similar images with opposite labels — this is the hardest
   problem to fix without better labels.

3. **No learning rate decay.** A fixed learning rate the whole time means
   the optimizer keeps taking big steps even when it should be settling. A
   cosine decay or step schedule would let the model converge more cleanly.

4. **No dropout or regularization.** The ResNet18 backbone has no dropout
   in the added head. The model can freely memorize training images.

## What would actually move the number

| Action | Expected impact | Difficulty |
|--------|----------------|------------|
| More labeled great sunsets (target: 1200+ positives) | High | Medium — requires labeling time |
| Early stopping (stop when val loss stops improving for 3 epochs) | Medium — prevents wasted training and captures true peak | Easy — small code change |
| Learning rate decay (cosine or step) | Medium — cleaner convergence | Easy — one optimizer change |
| Dropout on the classifier head | Medium — slows overfitting | Easy — one line |
| Better binary threshold (review which images are near the cutoff) | High — fixes label noise | Hard — requires manual review |
| Weighted random sampler + class weighting together | Low-medium — marginal | Easy — already supported |

---

## Recommended next experiment

Run a new config with:
- `epochs: 20` (more room to find the true peak)
- Add `EarlyStoppingPatience: 4` (stop if val loss does not improve for 4
  consecutive epochs) — requires a small train.py addition
- Add a cosine LR schedule — requires a small train.py addition
- Everything else the same as `v2_mild_crop_balanced`

This would give cleaner loss curves and tell you the true epoch ceiling
without manual tuning.

## How to regenerate these plots

```bash
# Regenerate all existing runs
python3 ml/plot_diagnostics.py --all

# Single run
python3 ml/plot_diagnostics.py \
  --run-dir ml/artifacts/experiments/20260314_070706_v2_mild_crop_balanced

# Compare two runs side by side
python3 ml/plot_diagnostics.py \
  --run-dir ml/artifacts/experiments/20260314_070706_v2_mild_crop_balanced \
  --run-dir ml/artifacts/experiments/20260314_082956_v2_no_crop_balanced
```

Plots save to `<run_dir>/plots/` and comparison saves to
`ml/artifacts/reports/comparison_<timestamp>.png`.

New experiments automatically generate plots at the end of the pipeline —
no extra steps needed.
